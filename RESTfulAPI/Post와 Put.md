
`OpenSearch`를 사용하다 보니 `index`를 만드는 API가 `PUT`으로 되어있는 것을 보았다. 그래서 오잉? 삽입인데 왜 `POST`가 아닌 `PUT`이지? 하면서 찾아 들어가다가 `POST`/`PUT`에 대해서 좀 더 잘 알게 되었다.

핵심은 이것이다.
  - URL: 어떤 리소스를 다루는가
  - Body: 그 리소스의 내용/설정은 무엇인가

  그래서
```http
PUT /code_chunks_demo
```
  는

- code_chunks_demo라는 이름의 인덱스 리소스를 대상으로 하고
- body는 그 인덱스의 설정과 매핑을 설명한다

즉 인덱스 이름은 body에 넣는 값이 아니라 리소스 식별자입니다.

---
  

게시판으로 비유하면 이렇게 보면 된다.

#### 1. 서버가 ID를 정하는 경우
```http
POST /posts
body:
{
	"title": "hello",
	"content": "world"
}
```
  이건 보통 서버가 글 번호를 생성한다.

#### 2. 클라이언트가 ID를 이미 아는 경우
```http
PUT /posts/123
body:
{
	"title": "hello",
	"content": "world"
}
```
이건 123이라는 리소스를 내가 지정해서 만들거나 덮어쓰는 것이다.
OpenSearch의 인덱스 생성은 이쪽에 더 가깝다.

```http
PUT /code_chunks_demo
```
즉, `code_chunks_demo`라는 이름의 인덱스를 만들어라”이다.

---
#### 3. 핵심 오해
› 그럼 `PUT`을 썻는데 만약에 리소스가 없을 때 삽입을 한다면 이게 `RESTful`한 `API`인가? `404` 코드를 반환하는게 맞지 않나? 하는 생각이 들었다. 그러나

>`PUT`으로 없던 리소스를 생성하는 건 `REST/HTTP` 의미상 이상한 게 아니라고 한다.
오히려 **클라이언트가 `URI`를 알고 있고, 그 `URI`의 상태를 완전히 지정할 때는 `PUT`**이 맞다.



나는 지금 이렇게 생각한 것이다.
```
- PUT = 수정
- POST = 생성
```
그런데 `HTTP` 의미론은 그렇게 단순하지 않다.

정확히는 이렇다.
- `POST`
  - 서버에게 “이 컬렉션 아래 새 리소스를 만들어줘”
  - 보통 서버가 ID/URI를 정함
- `PUT`
  - “이 URI의 리소스 상태를 이 body로 만들어라/바꿔라”
  - 보통 클라이언트가 URI를 이미 알고 있음

즉, **PUT은 “수정 전용”이 아닙니다.**
없으면 생성하고, 있으면 교체가 더 정확합니다.

› 야 그럼 이런것도되는거잔아 예를들어서 프론트에서 만약에 게시
판에들어간다라고하자. 상세보기또는 글쓰기 페이지를 같이써.
그리고나서 이 사람이 글쓸떄던 수정할떄던 걍 프론트에서 uuid
를 발급해. 그래서 이 uuid로 put을 요청해. 그럼어케되겟어
uuid 인 pk에 대해서 없으면 삽입 잇으면 수정 이런식이겟는데?
이게 되는거아냐?


• 네, 됩니다. 그리고 실제로도 그렇게 설계하는 시스템이 있습니
다.  
___헉...!!!___

네가 말한 흐름은 정확히 이런 겁니다.
- 프론트가 UUID 생성
- PUT /posts/{uuid}
- 서버는 {uuid} 리소스를 확인
- 없으면 생성
- 있으면 수정

이건 HTTP 의미론상 이상하지 않습니다.
이걸 보통 **client-assigned id + PUT upsert**라고 볼 수 있습니다.

## 이 방식이 성립하는 조건
1. 클라이언트가 리소스 식별자를 정할 수 있어야 함
	- 즉 uuid를 서버가 아니라 클라이언트가 가져도 되는 도메인인가
2. PUT이므로 전체 상태를 보내야 함
	- title만 보내고 일부만 바꾸는 식이면 PUT보다 PATCH가 맞습니다
3. create와 update를 같은 URI에서 처리해도 정책상 문제가 없어야 함

- 권한
- 감사 로그
- 상태 전이 규칙
## 그러면 응답은 보통 이렇게 갑니다

- 없어서 새로 생성됨: 201 Created
- 있어서 덮어씀: 200 OK 또는 204 No Content

즉 한 endpoint가 upsert 역할을 하는 겁니다.

———

## 그런데 “되는가”와 “좋은가”는 다릅니다

이 방식이 항상 좋은 건 아닙니다. 주의점이 있습니다.

### 1. create와 update는 보통 정책이 다릅니다

예:

- 생성은 누구나 가능
- 수정은 작성자만 가능

이런 경우 같은 PUT upsert로 뭉개면 정책이 흐려집니다.

### 2. PUT은 부분 수정용이 아닙니다

상세/수정 페이지를 같이 쓰는 건 괜찮지만, 수정 시 일부 필드만
보내면 의미론이 틀어집니다.

- 전체 교체면 PUT
- 일부 수정이면 PATCH

이걸 구분해야 합니다.

### 3. 서버가 ID를 통제하고 싶을 수 있습니다

- 정렬 가능한 ID
- 샤딩 전략
- 내부 추적 정책
- 보안/감사 요구

이런 게 있으면 POST /posts로 서버가 ID를 만드는 쪽이 더 낫습
니다.

### 4. “실수로 create”가 날 수 있습니다

수정이라고 생각하고 PUT /posts/{uuid} 했는데, 실제로는 없는
UUID면 새 글이 생겨버릴 수 있습니다.
이걸 막고 싶으면 조건부 요청이 필요합니다.

예:

- 생성만 허용: If-None-Match: *
- 수정만 허용: 현재 버전/ETag 검사

———

## 그래서 실무에서는 보통 이렇게 나뉩니다

### 흔한 공용 API
이유:

- 서버가 ID 관리
- 생성/수정 정책 분리 쉬움
- API 사용자가 덜 헷갈림

### 내부 시스템 / 오프라인 우선 / 동기화 시스템
- 생성/수정: PUT /posts/{client_generated_uuid}

이유:

- 클라이언트가 먼저 ID 필요
- 낙관적 UI
- 오프라인 작성 후 동기화
- idempotency 확보 쉬움

즉 네 아이디어는 이상한 게 아니라, 특정 상황에서는 아주 좋은
설계입니다.

---
› 그럼 저런걸 실제로 하는게 뭐가 있어?

• 많이 있습니다. 다만 주로 “저장소/동기화/리소스 관리” 성격의
API에서 많이 쓰고, 일반 게시판 비즈니스 API에서는 상대적으로
덜 씁니다.

대표 예시는 이렇습니다.

- OpenSearch / Elasticsearch
	- PUT /{index}/_doc/{id}
	- 문서 ID를 클라이언트가 정하면, 없으면 생성되고 있으면 교체

- Amazon S3
	- PUT /bucket/key 
	- 삽입하거나 덮어씀

- 오프라인 우선 앱에서 특히 잘 맞음.
- 내부 동기화 API
- 모바일/오프라인 앱에서 초안 작성 시 클라이언트가 `UUID`를 먼저 만들고, 서버에 `PUT /drafts/{uuid}`로 올리는 방식이 실제로 자주 쓰인다.

왜 이런 데서 잘 맞느냐면:
- 클라이언트가 식별자를 먼저 알아야 하고
- 같은 요청을 다시 보내도 결과가 같아야 하고
- “생성/수정”보다 “이 `URI`의 상태를 이 값으로 맞춘다”가 더 중
요하기 때문입니다.

반대로 게시판, 주문, 결제, 회원가입 같은 일반 비즈니스 API에
서는 보통 이쪽이 더 흔합니다.

- 생성: `POST /posts`
- 수정: `PATCH /posts/{id}` 또는 `PUT /posts/{id}`

이유는 간단합니다.
- 생성과 수정의 권한 정책이 다름
- 서버가 ID를 관리하고 싶음
- “수정하려던 요청이 실수로 생성”되는 걸 막고 싶음
- PUT은 원래 전체 교체 의미라서 부분 수정 UI와 잘 안 맞는 경우가 많음

즉 한 줄로 정리하면:

프론트가 UUID를 만들어 PUT으로 upsert하는 방식은 실제로 많이
쓰이지만, 주로 저장소/동기화/오프라인 우선 시스템에 잘 맞고,
일반 비즈니스 CRUD의 기본값은 아니다.

• 보통은 절대 사용자에게 “중복됐으면 취소하세요”를 1차 방어로 맡기지 않는다.
결제 중복은 서버와 결제사 연동이 막아야 한다.

핵심은 4단계입니다.
1. 프론트는 버튼 비활성화로 UX만 보조한다.
	- 이건 실수 감소용이다.
	- 정합성 보장은 아니다.
2. 서버는 idempotency key를 강제한다.
	- 결제 시도 1건마다 고유 키를 만든다.
	- 같은 키로 다시 오면 새 결제를 만들지 않고 기존 결과를 돌려준다.
3. DB에 유니크 제약을 둡니다.

- 예: merchant_order_id, payment_attempt_id, idempotency_key
- 서버 버그나 동시성 문제가 있어도 2중 생성이 마지막에 한 번
더 막히게 해야 합니다.

4. PG/결제사에도 같은 멱등 키를 전달합니다.
	- PayPal: PayPal-Request-Id
	- Adyen: idempotency-key
	- Stripe: Idempotency-Key
- 즉 우리 서버만 멱등하면 끝이 아니라, 외부 결제 호출도 멱등
해야 한다.

보통 흐름은 이렇다.
- 사용자가 결제 버튼 클릭
- 서버가 payment_attempt_id 생성 또는 받음. 
	- 모바일 네트워크 불안정, 프론트 재시도, 서버 재시도, 웹훅 지연도 다 원인이다. 그래서 “사용자에게 취소하게 하자”는 설계는 약하다.

- 버튼 비활성화: 해야 함
- 서버 멱등성: 필수
- DB 유니크 제약: 필수에 가까움
- 웹훅/비동기 상태확인: 필수에 가까움
- 사용자에게 취소 요청: 장애 복구의 마지막 수단일 뿐, 기본 전략 아님

공식 문서들도 이 방향이다.

- Stripe idempotent requests:

https://docs.stripe.com/api/idempotent_requests?create_charge-transfer_data-amount=&lang=go
- PayPal idempotency:

https://developer.paypal.com/reference/guidelines/idempotency/
- Adyen API idempotency:

https://docs.adyen.com/pt/development-resources/api-idempotency/
- HTTP Idempotency-Key 초안:

https://www.ietf.org/archive/id/draft-ietf-httpapi-idempotency-key-header-07.html

한 줄로 정리하면, 결제 중복은 “사용자 실수” 문제가 아니라 “멱등성 설계” 문제이다.
