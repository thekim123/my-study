## 🚀 클러스터 시작 순서 요약

1. 각 서버에 Kafka 압축 해제 (`tar -xzf`)
2. 위 설정대로 `config/kraft/server.properties` 수정
3. 각 서버에서 아래 명령 실행
    
```bash
# 각 서버에서 1번씩만 실행 (클러스터 ID는 동일하게 유지)
CLUSTER_ID=$(bin/kafka-storage.sh random-uuid)
bin/kafka-storage.sh format -t $CLUSTER_ID -c config/kraft/server.properties
```

> 위 명령에서 생성된 CLUSTER_ID는 모든 서버에서 동일하게 사용해야 합니다.

4. Kafka 시작:

```bash
bin/kafka-server-start.sh config/kraft/server.properties
```

## ✅ `controller.quorum.bootstrap.servers`는 **초기 부트스트랩 용도**
### 이 항목은 언제 사용되나요?

- Kafka **KRaft 모드 클러스터를 처음 시작할 때** 아직 `meta.properties`가 **없는 상태**에서 **Controller 노드끼리 서로를 찾기 위해** 사용

---

## 🔍 역할 요약

| 설정 키                                  | 역할                                                         |
| ------------------------------------- | ---------------------------------------------------------- |
| `controller.quorum.bootstrap.servers` | 메타데이터가 아직 없을 때, controller 간 통신을 위한 **부트스트랩 서버 목록**        |
| `controller.quorum.voters`            | 클러스터 구성 정보 (Controller node ID + IP + 포트). **실제 운영 시 사용됨** |

> 즉, `controller.quorum.bootstrap.servers`는 **클러스터 생성 전에만 의미가 있고**, `controller.quorum.voters`가 **항상 필요한 핵심 설정**

---

## 🎯 삭제해도 될까?

- 이미 `controller.quorum.voters`가 설정되어 있고 `storage format`을 진행해서 `meta.properties`가 생성되었으면

👉 **`controller.quorum.bootstrap.servers`는 삭제해도 됨**
Kafka 4.0.0 공식 가이드에서도 **`controller.quorum.voters`만 필수**로 표기하고 있고, `bootstrap.servers`는 선택 사항

---

## ✅ 결론

|상황|`controller.quorum.bootstrap.servers` 필요 여부|
|---|---|
|클러스터 포맷 전|선택적으로 사용 가능|
|클러스터 포맷 후 (`meta.properties` 존재)|❌ 제거해도 됨 (권장)|

---

불필요한 혼란 방지를 위해 프로덕션에서는 **`controller.quorum.voters`만 설정**하고 `bootstrap.servers`는 생략하는 걸 권장드려요. 필요하면 `meta.properties`가 없어도 동작하도록 디버깅용으로만 쓰는 정도입니다.
