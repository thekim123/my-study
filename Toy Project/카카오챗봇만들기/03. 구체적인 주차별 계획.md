좋다. 아래 계획은 **네가 PM/아키텍트 입장에서 팀을 통제할 수 있게 만든 실행 계획**이다.  
초점은 두 가지다.

첫째, **12주 안에 파일럿을 실제로 띄우는 것**.  
둘째, **각 역할군이 “뭘 했는지/안 했는지”를 금요일 리뷰에서 바로 판별할 수 있게 만드는 것**.

먼저 고정해야 할 기술 원칙부터 잡겠다.

OpenAI 쪽은 **Responses API**를 표준 인터페이스로 고정하고, 모델 출력은 **Structured Outputs**로 JSON schema를 강제한다. Structured Outputs는 JSON mode보다 강하게 schema adherence를 보장하고, Responses API는 OpenAI가 권장하는 통합 인터페이스다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/guides/migrate-to-responses/?utm_source=chatgpt.com "Migrate to the Responses API"))

이 프로젝트는 HR 평가 데이터가 오가므로, MVP는 **stateless Responses 호출**로만 구성하고 `store=false`를 기본값으로 둔다. OpenAI 문서상 Responses는 기본적으로 30일 application state를 보관하며, conversation objects와 그 아이템은 30일 TTL 대상이 아니고 별도 지속성이 생길 수 있으므로 이 프로젝트에서는 쓰지 않는 편이 안전하다. 또한 `safety_identifier`에는 해시된 사용자 식별자를 넣는 것이 권장된다. Zero Data Retention은 별도 eligibility가 필요한 옵션이다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/guides/your-data/?utm_source=chatgpt.com "Data controls in the OpenAI platform"))

모델 전략은 이렇게 고정한다. **writer는 `gpt-4.1-mini` 계열**, **hard-case planner는 `gpt-5.4`**, **임베딩은 `text-embedding-3-small`**이다. 이유는 현재 OpenAI 문서상 **SFT와 DPO가 명시적으로 지원하는 계열이 gpt-4.1 / gpt-4.1-mini / gpt-4.1-nano**이기 때문에, 주차 9~10의 fine-tuning까지 고려하면 writer를 처음부터 그 계열로 두는 편이 낫기 때문이다. 반면 GPT-5.4는 OpenAI가 기본 범용·복합 추론 모델로 권장하므로 hard-case planner에만 제한적으로 쓰는 것이 맞다. 임베딩은 `text-embedding-3-small`이 검색/유사도 용도로 설계되어 있고 기본 차원은 1536이다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "Supervised fine-tuning | OpenAI API"))

HR 원문 코퍼스는 OpenAI hosted File Search에 올리지 않는다. OpenAI의 hosted file search는 문서를 파싱·청크화하고 임베딩을 생성·저장하므로, 이 프로젝트에서는 **검색 인덱스는 내부 PostgreSQL/pgvector + GIN**으로 유지하고, 모델에는 최소한의 retrieved context만 넘기는 구조가 맞다. Fine-tuning용 파일도 OpenAI에 업로드되면 기본적으로 수동 삭제 전까지 남을 수 있으므로, 업로드·삭제 lifecycle을 별도 통제해야 한다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/assistants/tools/file-search/?utm_source=chatgpt.com "Assistants File Search | OpenAI API"))

또 하나. 계정/배포 운영에서 **rate limit tier**를 초반에 확인해야 한다. OpenAI는 usage tier별 제한을 문서화하고 있으므로, 주차 1에 DevOps와 PM이 테스트/파일럿 예상량 기준으로 한도 점검을 끝내야 한다. 프롬프트는 고정 prefix를 최대한 재사용하도록 설계해야 하며, OpenAI 문서상 Prompt Caching은 자동으로 적용되고 입력 비용과 지연을 크게 줄일 수 있다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/guides/rate-limits/?utm_source=chatgpt.com "Rate limits | OpenAI API"))

**매주 수요일**
- 30분 기술 checkpoint
- blocker가 있으면 이때가 아니라 **발생 당일** 올렸어야 함
    
**매주 금요일**
- 말로 보고 금지
- 반드시 **실행 화면 / API 응답 / SQL 결과 / notebook 결과 / PR 링크** 중 하나를 보여줘야 함
- “거의 됐다”, “로컬에선 된다”, “데이터가 이상해서”는 완료로 인정하지 않음
    
**모든 티켓의 완료 정의**
- 코드
- 테스트
- 문서
- 데모
- 로그/에러 처리
- owner와 reviewer 명시

**이 프로젝트에서 금지할 변명**
- “프롬프트가 아직 감이라서”
- “데이터가 복잡해서”
- “UI는 나중에 붙이면 돼서”
- “일단 모델부터 보고”
- “운영은 나중에 생각해도 돼서”
    
이 프로젝트는 **snapshot → retrieval → generation → validation → human approval** 체인이 모두 연결되어야 값이 나온다. 하나라도 빠지면 그냥 실험이다.

## 3. 비기능 목표와 Gate
이 수치는 네가 기준선으로 박아도 된다.

**주차 6 gate**
- 실제 case_id로 end-to-end 생성/수정/승인 가능
- public/private 분기 정상 동작
- 추천안 2개, 근거, 위험 플래그가 UI에 표시됨
    
**주차 8 gate**
- leakage rate = 0
- required term coverage ≥ 95%
- grounding rate ≥ 90%
- p95 latency ≤ 6초
- hard-case planner 사용 비율 ≤ 15%
    
**주차 10 gate**
- fine-tuned writer가 baseline보다
    - acceptance rate 개선 또는
    - normalized edit distance 감소  
        둘 중 하나는 명확히 보여야 함
        
**주차 12 gate**
- 20~30명 파일럿 사용자에게 배포 가능
- rollback/runbook 존재
- backup/restore 검증 완료
- DPO pair 수집 시작

## 6. 이 프로젝트에서 절대 순서를 바꾸면 안 되는 것

1. **snapshot과 ACL을 먼저 고정**

- 주차 6 전에 UI polish 집착
    
- 주차 8 전에 모델 비교 놀이
    
- 주차 9 전에 데이터셋 governance 없이 업로드

## 7. 네가 팀에 바로 내릴 수 있는 한 줄 지시문

각 역할에 이 문장을 그대로 주면 된다.

**PM/아키텍트**  
“매주 금요일에 scope, risk, decision log를 업데이트하고, freeze되지 않은 항목은 다음 단계 작업을 막아라.”

**BE/LLM-1**  
“case_id 하나를 넣었을 때 deterministic snapshot과 scope-safe retrieval을 반환하는 API를 주차 4 안에 끝내라.”

**BE/LLM-2**  
“주차 6 안에 Structured Outputs 기반 writer와 validator를 붙여 실제 승인 플로우가 돌게 만들어라.”

**프런트엔드**  
“주차 6 안에 reviewer가 추천안 2개를 보고 근거와 위험을 확인한 뒤 edit/approve/reject 할 수 있게 만들어라.”

**데이터/ML**  
“주차 4 안에 300k 코퍼스 정제를 끝내고, 주차 10 안에 20k~30k curated SFT dataset과 FT 비교 리포트를 내라.”

**DevOps**  
“주차 2 안에 dev/stg/prod와 SSO를 열고, 주차 12 안에 rollback·backup·monitoring을 실제로 검증해라.”

**HR SME**  
“주차 2 안에 용어집·금지표현·rubric을 확정하고, 주차 10까지 blind eval로 FT 개선 여부를 판정해라.”
