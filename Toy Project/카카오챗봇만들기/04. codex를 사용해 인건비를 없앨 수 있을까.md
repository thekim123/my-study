가능하다. 다만 **“사람 5명을 Codex로 대체”**로 보면 안 된다.  
최고의 제품을 만들려면 **작은 인간 핵심팀이 방향·정책·품질 기준을 잡고, Codex를 병렬 구현 에이전트 군으로 쓰는 방식**이 맞다. Codex는 코드 작성, 버그 수정, 코드베이스 이해, PR 제안 같은 소프트웨어 작업을 수행할 수 있고, CLI는 로컬 터미널에서 현재 디렉터리의 코드를 읽고 수정하고 명령을 실행할 수 있다. 반면 Codex Cloud는 자체 클라우드 환경에서 백그라운드로 병렬 작업을 돌리는 쪽에 가깝다. Codex 앱은 병렬 스레드, Git 기능, worktree, automations를 제공한다. ([OpenAI](https://openai.com/index/introducing-codex/?utm_source=chatgpt.com "Introducing Codex"))

내 가장 강한 권고는 이거다.

**Codex를 “개발자 대체재”가 아니라 “구현 속도를 3~5배 올리는 실행기”로 써라.**  
특히 네 프로젝트는 HR 평가, 공개/비공개 권한, 도메인 용어, SFT/DPO, 감사 로그까지 걸려 있어서, 제품의 승부는 코드량이 아니라 **판단 경계**에서 난다. 이 경계는 사람이 잡아야 한다.

## 무엇은 사람이 직접 하거나 반드시 고용해야 하나

아래 네 가지는 **반드시 인간 책임자**가 있어야 한다.

첫째, **제품/아키텍처 오너**다.  
요구사항을 고정하고, “무엇을 하지 않을지”, “public/private를 어디서 분리할지”, “어떤 지표로 출시 여부를 판단할지”를 결정하는 사람이다. 이건 Codex에 맡기면 안 된다.

둘째, **HR SME**다.  
좋은 코멘트/나쁜 코멘트의 기준, 금지 표현, 회사 용어, 공개/비공개 경계, 선호 어투 기준은 사람 판단이 필요하다. DPO나 SFT도 결국 사람이 승인한 기준 위에 서야 한다. DPO는 주관적 선호를 반영하는 방식으로 공식 문서에 설명되어 있고, 톤과 스타일 같은 주관적 영역에 적합하다. ([OpenAI 개발자 포털](https://developers.openai.com/api/docs/guides/direct-preference-optimization/?utm_source=chatgpt.com "Direct preference optimization | OpenAI API"))

셋째, **실제 코드 오너가 되는 시니어 엔지니어**다.  
네가 직접 이 역할을 할 수 있으면 고용하지 않아도 된다. 하지만 이 역할 없이 Codex만으로 가면, 설계 경계가 무너지고 구현이 산개할 가능성이 높다.

넷째, **DevOps/보안 오너**다.  
SSO, secrets, audit log, backup/restore, 배포 롤백, 권한 분리, OpenAI 키 사용 정책은 “Codex가 작성할 수 있는 영역”이지 “Codex가 책임질 수 있는 영역”이 아니다.

즉, 최소 핵심 인력은 이렇게 보는 게 맞다.

- 네가 기술 리드까지 할 수 있으면: **너 + HR SME + DevOps/보안 fractional**
    
- 네가 기술 리드를 못 하면: **너 + 시니어 백엔드/LLM 리드 1명 + HR SME + DevOps/보안 fractional**
    

그 외 구현량은 Codex로 크게 줄일 수 있다.

## 무엇은 Codex에 강하게 맡겨도 되나

이건 Codex-first로 가도 된다.

- FastAPI 엔드포인트 구현
    
- Pydantic/SQLAlchemy 모델
    
- snapshot builder의 보일러플레이트
    
- retrieval SQL/ORM 초안
    
- validator 코드 초안
    
- Next.js 화면 구현
    
- 관리자 UI
    
- 데이터 정제 스크립트
    
- 임베딩 백필 배치
    
- eval harness 초안
    
- Terraform 모듈 초안
    
- 테스트 코드
    
- 문서 초안
    
- PR 설명문과 체크리스트 작성
    

Codex는 이런 반복적 개발 작업, 리팩터링, 테스트, 셋업 작업에 적합하고, 공식 best practices도 Codex를 일회성 어시스턴트보다 **설정하고 개선해 나가는 팀원**처럼 다루라고 권장한다. 또 `AGENTS.md`로 지속 지침을 주고, 반복 작업은 skills와 automations로 묶는 패턴을 권장한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))

## 그래서 운영 모델은 어떻게 가져가야 하나

가장 좋은 모델은 **Human Core + Codex Swarm**이다.

사람은 세 종류다.
- **의사결정자**: 너
- **코드 품질 오너**: 너 또는 시니어 엔지니어
- **정책 오너**: HR SME
- **운영 오너**: DevOps/보안

Codex는 역할별로 병렬 스레드를 돌린다.
- snapshot/DB 구현 에이전트
- retrieval/검색 에이전트
- writer/validator 에이전트
- 프런트엔드 에이전트
- 데이터/eval 에이전트
- 리뷰/테스트 에이전트

Codex 앱은 여러 작업을 병렬 스레드와 worktree로 관리하는 데 맞고, Codex CLI는 로컬 머신에서 코드 읽기·수정·명령 실행을 하는 데 맞다. 그래서 **민감한 사내 리포지토리에는 CLI/IDE를 기본**, **병렬 백그라운드 작업은 앱/Cloud를 선택적으로** 쓰는 구성이 자연스럽다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/cli/?utm_source=chatgpt.com "Codex CLI"))

## 네 프로젝트에서 가장 좋은 실행 방식

나는 처음부터 **“Codex를 프로그램으로 오케스트레이션”**하지 않겠다.  
즉, 첫 단계에서 Agents SDK나 복잡한 멀티에이전트 프레임워크부터 짜지 않겠다.

처음에는 이렇게 간다.
1. **Codex CLI/IDE + Codex 앱**
2. **루트 `AGENTS.md`**
3. **`.codex/config.toml`**
4. **repo-local skills**
5. **강한 PR 규칙**
6. **사람 승인 후 머지**

이유는 간단하다. OpenAI 공식 best practices도 Codex는 **먼저 Plan mode, AGENTS.md, config, skills, automations**로 운영 습관을 잡으라고 설명한다. 복잡한 자동화는 안정된 반복 작업부터 붙이는 것이 맞다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))

## 리포지토리를 Codex 친화적으로 바꾸는 것이 첫 일이다

너는 프로젝트 시작 전에 리포지토리에 아래를 반드시 넣어야 한다.

### 1. 루트 `AGENTS.md`

이 파일이 사실상 **Codex용 팀 운영 헌법**이다.  
OpenAI 문서에 따르면 Codex는 작업 전에 `AGENTS.md`를 읽고, 전역/프로젝트/하위 디렉터리 수준의 지침을 계층적으로 적용한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/guides/agents-md/?utm_source=chatgpt.com "Custom instructions with AGENTS.md"))

여기에는 최소한 아래가 있어야 한다.
- 시스템 개요
- 디렉터리 구조
- 빌드/테스트/린트 명령
- 코딩 규칙
- 금지 사항
- 보안 규칙
- Definition of Done
- 이 프로젝트 특유의 규칙
    - public/private 분리
    - OpenAI 호출 시 `store=false`
    - snapshot 외 raw join dump 사용 금지
    - validator 없는 writer 코드 금지
    - SFT 데이터는 승인된 curated set만 사용
        

### 2. `.codex/config.toml`

공식 best practices는 Codex 동작 일관성을 위해 config를 조기에 잡으라고 권장한다. repo-scoped `.codex/config.toml`은 프로젝트별 동작을 고정하는 데 쓴다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))

여기서는 최소한 아래를 고정해라.

- 기본 모델
    
- sandbox/approval 정책
    
- 작업 루트
    
- profile
    
- MCP/도구 연결
    
- reasoning effort 기본값
    

### 3. repo-local skills

skills는 `SKILL.md`와 보조 스크립트/레퍼런스를 묶어서 반복 작업을 더 안정적으로 수행하게 한다. 공식 문서상 skills는 CLI, IDE, 앱에서 공통으로 쓸 수 있고, 필요한 순간에만 전체 지침이 로드되어 컨텍스트를 아낀다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/skills/?utm_source=chatgpt.com "Agent Skills"))

너 프로젝트용으로는 적어도 아래 여섯 개를 만들겠다.

- `snapshot-contract`
    
- `acl-safety-check`
    
- `retrieval-query-build`
    
- `writer-validator-contract`
    
- `eval-runner`
    
- `ft-dataset-curation`
    

이렇게 해두면 사람의 말이 아니라 **레포가 Codex를 훈련**한다.

## 실제 티켓 수행 방식

각 티켓은 사람이 직접 쓰되, 티켓 형식을 강제해야 한다.  
Codex는 “막연한 요구사항”보다 “완료 기준이 명확한 작업”에서 훨씬 잘 동작한다. Codex 공식 문서도 큰 저장소나 고위험 작업일수록 **명확한 컨텍스트와 done 정의**가 중요하다고 설명한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))

티켓 템플릿은 이렇게 잡아라.

- 배경
    
- 목표
    
- 수정 가능한 파일/디렉터리 범위
    
- 수정 금지 범위
    
- 입출력 계약
    
- 테스트 조건
    
- 보안/권한 제약
    
- 완료 정의
    
- 비목표
    

그리고 각 티켓은 항상 이 흐름으로 돈다.

1. **Codex에게 먼저 계획만 시킨다.**  
    OpenAI 공식 best practices도 복잡하고 애매한 작업은 먼저 계획하라고 권장한다. Plan mode를 쓰거나 먼저 질문하게 하라고 한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))
    
2. **구현용 Codex 스레드 하나**
    
3. **리뷰/테스트용 Codex 스레드 하나**
    
4. **사람이 최종 diff 선택**
    
5. **사람이 merge 승인**
    

즉, 한 사람 + Codex 2개 스레드가 **1인 개발자 + 1인 리뷰어 + 1인 QA**처럼 작동하게 만드는 것이다.

## 역할별로 Codex를 어떻게 붙일까

### 백엔드/LLM

가장 Codex 효율이 좋다.  
snapshot builder, API, validator, OpenAI wrapper, Pydantic schema, SQLAlchemy 모델, batch jobs, tests는 Codex가 강하게 밀 수 있다. 대신 **아키텍처 경계, API 계약, 모델 선택, 데이터 보안 규칙**은 사람이 잡아야 한다.

### 프런트엔드

화면 구현, 상태 처리, diff UI, 관리자 화면은 Codex를 많이 쓸 수 있다. 최근 OpenAI도 Codex app 중심으로 Figma 연계와 UI 개발 사례를 소개하고 있다. 다만 제품 품질을 끌어올리려면 결국 **사람의 UX 판단**이 필요하다. ([OpenAI 개발자 포털](https://developers.openai.com/blog/building-frontend-uis-with-codex-and-figma/?utm_source=chatgpt.com "Building frontend UIs with Codex and Figma"))

### 데이터/ML

ETL, dedupe, embedding backfill, eval runner, notebook 초안, training dataset export는 Codex가 잘 도와준다. 하지만 **gold set 기준 정의, inclusion/exclusion 규칙, SFT/DPO 데이터 승인**은 반드시 인간이 해야 한다. DPO와 SFT는 데이터가 나쁘면 모델이 더 일관되게 나빠진다.

### DevOps

Terraform, Helm, GitHub Actions, CloudWatch 대시보드, rollback 스크립트 초안은 Codex가 잘 한다. 하지만 **secret management, IAM, SSO, 백업/복구, production access**는 인간 오너가 필요하다.

### PM/아키텍트

이건 Codex 대체가 거의 안 된다.  
Codex는 문서 초안과 옵션 비교안은 낼 수 있지만, 요구사항을 고정하고 tradeoff를 감수하는 책임은 못 진다.

### HR SME

대체 불가다.  
이 역할은 “라벨러”가 아니라 **정책과 품질의 근원**이다.

## 사람을 줄인다면 어디를 줄이고 어디를 남겨야 하나

가장 현실적인 축소안은 이거다.

**남겨야 하는 인간**

- 너: PM/아키텍트
    
- 시니어 백엔드/LLM 리드 1명
    
    - 네가 이걸 할 수 있으면 생략 가능
        
- HR SME 0.5
    
- DevOps/보안 0.3~0.5
    

**Codex로 강하게 대체 가능한 영역**

- 두 번째 백엔드/LLM 엔지니어 대부분
    
- 프런트엔드 구현량 상당 부분
    
- 데이터/ML의 스크립팅/실험/정제 작업 상당 부분
    

즉, 네가 기술적으로 강하면 **원래 5명 수준의 구현량을 2~3명의 인간 + Codex 병렬 에이전트**로 충분히 밀 수 있다. 다만 그 전제는 **네가 PR 리뷰와 설계 판단을 실제로 할 수 있을 때**다.

## 아주 중요한 경계선

이 네 가지는 Codex에 맡기면 안 된다.

- main/prod 자동 머지
    
- public/private ACL 결정
    
- SFT/DPO 데이터셋 승인
    
- go-live 승인
    

이건 전부 **사람 서명 절차**가 있어야 한다.

## 네가 지금 당장 해야 할 최선의 시작 방식

첫 주는 개발보다 **Codex 운영체제**를 깔아라.

1. Codex CLI와 앱을 설치하고, 기본 코딩 모델은 Codex 문서 권장대로 **`gpt-5.4`**부터 시작한다. OpenAI 공식 문서는 “대부분의 Codex 작업은 gpt-5.4로 시작하라”고 안내한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/models/?utm_source=chatgpt.com "Codex Models"))
    
2. 루트 `AGENTS.md`를 만든다. `/init`로 뼈대를 만들 수 있지만, 반드시 네 프로젝트 규칙으로 수정해야 한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/learn/best-practices?utm_source=chatgpt.com "Best practices"))
    
3. `.codex/config.toml`을 만든다.
    
4. 위에서 말한 6개 skills를 만든다.
    
5. PR template과 branch protection을 만든다.
    
6. “한 티켓당 계획 스레드 1개 + 구현 스레드 1개 + 리뷰 스레드 1개” 운영을 시작한다.
    
7. 안정된 반복 업무만 automation으로 올린다. Codex 공식 문서도 automations는 skills와 함께 유지보수 가능한 형태로 쓰라고 설명한다. ([OpenAI 개발자 포털](https://developers.openai.com/codex/app/automations/?utm_source=chatgpt.com "Automations"))
    

## 그리고 네 제품 모델 쪽은 따로 생각해야 한다

Codex는 **개발 인력 대체/보조 도구**이고, 네 제품 안에 들어갈 writer 모델은 별개다.  
네 제품은 계속 **Responses API + Structured Outputs + 내부 retrieval** 구조로 가고, writer는 이후 **`gpt-4.1-mini` 기반 SFT/DPO**로 고정하는 편이 좋다. OpenAI 공식 문서상 `gpt-4.1-mini`는 Structured Outputs와 fine-tuning을 지원하고, DPO도 `gpt-4.1-mini-2025-04-14`를 포함한 4.1 계열에서 지원된다.
