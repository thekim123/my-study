네. 아래는 **“실제로 발주하고 구현하는 기준안”**으로 잡은 계획이다.  
기준은 이렇게 고정하겠다.

- 배포 환경: **AWS Seoul(ap-northeast-2)** 중심
    
- 모델 연동: **OpenAI Responses API**
    
- 검색 저장소: **Amazon RDS for PostgreSQL 16 + pgvector + PostgreSQL full-text search**
    
- 백엔드: **Python 3.12 + FastAPI + Pydantic v2 + SQLAlchemy**
    
- 비동기 작업: **Celery + Redis**
    
- 프런트: **Next.js + TypeScript**
    
- IaC/배포: **Terraform + GitHub Actions**
    
- 운영 모드: **추천 + 인간 승인(HITL), 자동 확정 금지**
    

이 기준이 맞는 이유는 단순하다. OpenAI는 지금 **Responses API**를 중심으로 가고 있고, Structured Outputs로 JSON Schema 강제를 지원한다. SFT와 DPO는 둘 다 **gpt-4.1 계열**에서 가능하며, 임베딩은 `text-embedding-3-small`이 검색용으로 충분히 싸고 다국어 성능이 좋다. PostgreSQL은 원래 full-text search를 지원하고, pgvector는 Postgres 안에서 벡터 검색을 붙일 수 있으며, RDS for PostgreSQL도 pgvector를 지원한다. AWS Fargate, RDS, ElastiCache는 Seoul 리전에서 사용 가능하다. ([OpenAI Developers](https://developers.openai.com/api/docs/assistants/migration/ "https://developers.openai.com/api/docs/assistants/migration/"))

## 0. 먼저 통과해야 할 의사결정 3개

이 프로젝트는 시작 전에 아래 3개가 확정되어야 한다.

첫째, **외부 LLM API에 HR 텍스트를 보내는 것이 법무/보안상 허용되는지**를 확인해야 한다. 허용된다면 OpenAI 연동안을 유지하고, 허용되지 않으면 이 설계는 모델 계층부터 바뀐다. 둘째, HR 데이터라면 **Responses API의 기본 30일 application state 보관 정책**을 그대로 쓸지, 아니면 `store=false` 또는 Zero Data Retention 요건으로 갈지를 정해야 한다. 셋째, API 요청마다 **해시된 `safety_identifier`**를 넣어 사용자 단위 안전 추적이 가능하게 해야 한다. OpenAI는 기본적으로 Responses API에 30일 보관이 있고, ZDR이면 `store=false`처럼 취급된다고 설명하며, 안전 식별자는 해시된 username/email 같은 값으로 넣으라고 권장한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/your-data/ "https://developers.openai.com/api/docs/guides/your-data/"))

내 권장은 이거다.  
**HR 본문은 비식별화 후 전송, `store=false` 기본, reviewer/세션 단위 `safety_identifier` 사용**이다. 이걸 못 하면 프로젝트를 멈추는 게 맞다.

## 1. 최종 아키텍처

이 시스템은 모델이 중심이 아니라 **데이터 계약과 검증 파이프라인**이 중심이다. 내가 실제로 만들면 아래 8개 서비스로 나눈다.

1. **Snapshot Service**  
    원천 평가 테이블들을 조인해서 `evaluation_case_snapshot` 생성
    
2. **ACL/Policy Filter**  
    public 요청이면 private 필드 자체를 제거  
    reviewer 권한 밖 데이터 차단
    
3. **Retrieval Service**  
    세 개의 인덱스 조회
    
    - 유사 평가 사례
        
    - 회사 용어/정책
        
    - 작성자/조직 스타일 예문
        
4. **Generation Orchestrator**  
    hard case 여부 판단 후 planner/writer 분기
    
5. **Writer Model Service**  
    Structured Output으로 추천안 2개 생성
    
6. **Validator Service**  
    용어, 스코프, 근거, 표현 금지, 점수-문장 정합성 검사
    
7. **Reviewer UI**  
    추천안, 근거, 위험 플래그, 수정 UI 제공
    
8. **Feedback/Dataset Builder**  
    승인/수정/거절 로그를 SFT/DPO 학습셋으로 적재
    

Structured Outputs는 JSON Schema 강제를 지원하므로, 추천안·근거·위험플래그를 자유문장이 아니라 구조화 객체로 받는 데 적합하다. OpenAI의 가이드도 fine-tuning 전에 eval을 세팅하고, production에서는 structured output, latency optimization, safety/human oversight를 함께 설계하라고 본다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/structured-outputs/?utm_source=chatgpt.com "Structured model outputs | OpenAI API"))

## 2. 기술 스택: 그대로 고정해서 가면 된다

### 애플리케이션 계층

프런트는 **Next.js + TypeScript + TanStack Query + shadcn/ui**로 간다. 내부 업무툴에서는 폼, 승인, 비교, diff UI가 많아서 React 생태계가 유리하다.  
백엔드는 **FastAPI + Pydantic v2 + SQLAlchemy 2.x**로 간다. 이유는 snapshot 생성, 데이터 전처리, LLM 오케스트레이션, offline eval, fine-tuning dataset build를 전부 Python으로 묶기 쉽기 때문이다.

### 비동기/배치

**Celery + Redis**로 간다. 작업은 아래만 분리한다.

- snapshot 재생성
    
- 임베딩 생성
    
- retrieval 재색인
    
- eval 배치
    
- SFT/DPO 데이터셋 빌드
    
- 관리자 재생성 요청
    

### 데이터 저장소

**RDS for PostgreSQL 16 + pgvector + PostgreSQL full-text search**로 간다.  
이유는 30만 건 규모에서는 별도 vector DB를 두지 않아도 된다. Postgres는 원래 `tsvector/tsquery` 기반 full-text search와 GIN/GiST 인덱스를 지원하고, pgvector는 exact/approximate nearest neighbor search를 지원한다. RDS for PostgreSQL도 pgvector를 지원한다. 이 조합이면 **metadata filter + keyword search + vector similarity**를 한 DB 안에서 처리할 수 있다. ([PostgreSQL](https://www.postgresql.org/docs/current/textsearch.html "https://www.postgresql.org/docs/current/textsearch.html"))

### 인프라

AWS는 아래로 고정한다.

- **ECS Fargate**: API/worker 컨테이너
    
- **ALB**: 웹 트래픽
    
- **RDS PostgreSQL**: 시스템 DB + hybrid retrieval
    
- **ElastiCache Redis**: queue/cache
    
- **S3**: 원본 추출물, eval artifacts, training dataset
    
- **CloudWatch**: 로그/메트릭
    
- **KMS + Secrets Manager**: 암호화/비밀
    
- **AWS WAF**: 내부망 노출/외부 접근 정책 있으면 추가
    
- **OIDC SSO**: 사내 IdP(Okta/Azure AD/Google Workspace) 연동
    

Fargate는 서버 관리가 필요 없고, RDS는 백업/복구/보안 패치를 관리해준다. RDS는 instance-hour, storage, backup, data transfer 등으로 과금되며 per-second billing을 지원한다. ALB는 시간 요금과 LCU 기준 과금이다. ([Amazon Web Services, Inc.](https://aws.amazon.com/rds/postgresql/pricing/ "https://aws.amazon.com/rds/postgresql/pricing/"))

## 4. 검색 설계: 인덱스를 3개로 분리한다

### A. 유사 평가 사례 인덱스

과거 2차 평가 최종 승인 코멘트를 넣는다.  
검색 키는 단순 텍스트가 아니라 아래 조합이다.

- role family
    
- level
    
- scope
    
- score pattern
    
- peer/1차/2차 간 signal divergence
    
- mission type
    
- org level
    

검색 점수는 이렇게 두겠다.

`final_score = 0.55 * vector + 0.20 * fulltext + 0.15 * metadata_match + 0.10 * score_pattern_similarity`

### B. 정책/용어 인덱스

회사 핵심가치, 금지 표현, 권장 표현, 조직별 목표 설명, 평가 가이드를 넣는다.  
정확한 용어가 중요하므로 semantic only가 아니라 **keyword + semantic hybrid**로 간다. PostgreSQL full-text search와 pgvector 조합이 여기 맞다. ([PostgreSQL](https://www.postgresql.org/docs/current/textsearch.html "https://www.postgresql.org/docs/current/textsearch.html"))

### C. 스타일 인덱스

개인 스타일을 그대로 복제하지 말고, 아래 우선순위로 쓴다.

1. reviewer 개인 스타일: 과거 승인 코멘트 **50건 이상**일 때만 사용
    
2. 조직 스타일: 팀/직군 pool **200건 이상**일 때 사용
    
3. 회사 기본 스타일: fallback
    

이 규칙은 강제한다.  
개인 데이터가 적은데 개인 어투를 흉내 내면 과적합과 어색한 문장이 늘어난다.

## 5. 요청 처리 플로우: 이대로 구현하면 된다

`POST /v1/reviews/{case_id}/recommendations:generate?scope=private`

이 API는 아래 순서로만 돈다.

1. `case_id`로 최신 snapshot 조회
    
2. scope에 맞는 JSON view만 로드
    
3. retrieval 3종 수행
    
4. hard-case router 판정
    
5. writer 호출
    
6. validator 실행
    
7. UI에 추천안 2개 + 근거 + risk flags 반환
    
8. reviewer가 수정/승인/거절
    
9. 로그 적재
    
10. nightly dataset builder가 SFT/DPO 데이터 생성
    

### hard-case router 규칙

아래 중 하나면 planner를 추가 호출한다.

- 미션 수 5개 초과
    
- peer vs 1차 점수 차이 평균 1.0 이상
    
- private comment가 3개 이상
    
- 금지 표현 위험 플래그 존재
    
- retrieval top-3 유사도 평균이 낮음
    

### 모델 호출 순서

기본 경로는 **모델 1회 호출**만 쓴다.  
hard case만 planner 1회 + writer 1회를 쓴다.

이게 중요한 이유는 비용보다도 **지연시간** 때문이다. OpenAI는 latency를 줄이려면 요청 수와 토큰 수를 줄이고, 고정 프롬프트를 재사용해서 prompt caching 이점을 얻으라고 권장한다. Prompt Caching은 반복 prefix가 있을 때 input token 비용을 최대 90%, latency를 최대 80%까지 줄일 수 있다고 설명한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/latency-optimization/ "https://developers.openai.com/api/docs/guides/latency-optimization/"))

## 6. 모델 선택: 지금은 이 조합이 제일 현실적이다

### 운영 writer

**`gpt-4.1-mini-2025-04-14` 기반 fine-tuned writer**로 간다.  
이유는 SFT와 DPO가 둘 다 지원되고, 가격이 낮고, output contract를 고정하기 좋기 때문이다. OpenAI 문서상 SFT와 DPO는 `gpt-4.1`, `gpt-4.1-mini`, `gpt-4.1-nano` 계열을 지원한다. `gpt-4.1-mini` 가격은 input $0.40 / cached input $0.10 / output $1.60 per 1M tokens다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "https://developers.openai.com/api/docs/guides/supervised-fine-tuning/"))

### hard-case planner

**`o4-mini`**를 hard case에만 사용한다.  
reasoning이 필요한 케이스만 쓰고, default path에는 넣지 않는다. `o4-mini` 가격은 input $1.10 / cached input $0.275 / output $4.40 per 1M tokens다. ([OpenAI Developers](https://developers.openai.com/api/docs/pricing/ "https://developers.openai.com/api/docs/pricing/"))

### embeddings

**`text-embedding-3-small`**를 쓴다.  
가격은 $0.02 / 1M tokens이고, OpenAI는 이 모델을 lower cost, higher multilingual performance라고 설명한다. 한국어 평가문장 검색에는 이 정도면 충분하다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/embeddings/?utm_source=chatgpt.com "Vector embeddings | OpenAI API"))

### 구현 원칙

- **Assistants API로 새로 만들지 말고 Responses API**로 만든다.
    
- **free-form text가 아니라 Structured Output**을 강제한다.
    
- **모델 alias가 아니라 snapshot model ID**를 써서 재현성을 확보한다.
    

OpenAI는 Assistants API를 deprecated 상태로 두었고 2026-08-26 종료 예정이라고 안내한다. ([OpenAI Developers](https://developers.openai.com/api/docs/assistants/migration/ "https://developers.openai.com/api/docs/assistants/migration/"))

## 7. 출력 계약(JSON Schema)

writer가 바로 문장만 반환하면 안 된다.  
이 스키마로 강제한다.

```json
{
  "scope": "private",
  "drafts": [
    {
      "text": "string",
      "tone": "balanced|coaching|direct",
      "required_terms_used": ["string"],
      "evidence_refs": ["peer:ownership", "manager:execution"],
      "risk_flags": ["none|possible_leakage|weak_grounding|missing_term"]
    },
    {
      "text": "string",
      "tone": "balanced|coaching|direct",
      "required_terms_used": ["string"],
      "evidence_refs": ["peer:collaboration"],
      "risk_flags": []
    }
  ],
  "reasoning_summary": {
    "strengths": ["string"],
    "improvement_points": ["string"]
  }
}
```

Structured Outputs는 supplied JSON Schema를 따르게 보장하는 기능이므로, 자유문장보다 이런 형식이 훨씬 운영하기 쉽다. validator, UI, audit log, eval 전부 이 구조 위에 올라간다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/structured-outputs/?utm_source=chatgpt.com "Structured model outputs | OpenAI API"))

## 8. validator: 코드 기반으로 먼저 막고, LLM judge는 나중에 최소화한다

validator는 6개다.

1. **SchemaValidator**  
    JSON Schema 통과 여부
    
2. **ScopeLeakValidator**  
    public 응답에 private source/evidence가 들어갔는지 검사
    
3. **TermValidator**  
    필수 용어 누락, 금지 표현 포함 여부 검사
    
4. **EvidenceValidator**  
    `evidence_refs`가 실제 snapshot/retrieval source에 존재하는지 검사
    
5. **ConsistencyValidator**  
    낮은 점수인데 과도하게 긍정적 표현인지 룰 기반 검사
    
6. **ToneValidator**  
    reviewer/org/company style hierarchy 위반 여부 검사
    

여기서 1~5는 **코드로** 처리하고, 정말 애매한 grounding만 LLM judge를 쓴다.  
OpenAI function/tool usage 가이드와 production best practices도 “모델이 할 일을 줄이고 가능한 것은 코드로 내리라”는 방향이다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/function-calling/ "https://developers.openai.com/api/docs/guides/function-calling/"))

## 9. fine-tuning 계획: 반드시 이 순서로만 한다

### Phase 1: Baseline without FT

먼저 **RAG + structured output + validator**만 만든다.  
이걸로 acceptance rate와 edit distance를 측정한다.

### Phase 2: SFT

그 다음 SFT를 붙인다. 목적은 지식 추가가 아니라 **출력 계약 고정**이다.  
OpenAI도 SFT를 “desired style and content를 더 reliably produces” 하게 만드는 방식으로 설명하고, fine-tuning 전에 eval을 먼저 세팅하라고 권장한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "https://developers.openai.com/api/docs/guides/supervised-fine-tuning/"))

#### SFT 데이터 규칙

- 처음부터 30만 건 전부 넣지 말 것
    
- **20k~50k 고품질 승인 코멘트**부터 시작
    
- 입력 포맷은 **실제 inference와 동일한 snapshot + retrieved context 구조**
    
- 너무 짧은 코멘트, 형식적 복붙, policy 위반 문장은 제외
    

OpenAI는 적은 수의 좋은 예제로 먼저 시작해 보고 eval 결과를 본 뒤 늘리라고 권장한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "https://developers.openai.com/api/docs/guides/supervised-fine-tuning/"))

### Phase 3: DPO

DPO는 **운영 로그가 쌓인 뒤** 붙인다.  
조건은 아래 둘 중 하나다.

- 5k 이상 preference pair 확보
    
- reviewer edit log에서 accepted vs edited/rejected를 안정적으로 추출 가능
    

DPO는 `prompt + preferred_output + non_preferred_output` 형식이며, tone/style 같은 주관적 선호를 학습하는 데 적합하다고 OpenAI가 설명한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/direct-preference-optimization/ "https://developers.openai.com/api/docs/guides/direct-preference-optimization/"))

#### DPO 데이터 규칙

- preferred: 최종 승인 코멘트
    
- non-preferred: 초기 추천안, rejected 후보, 수정 전 초안
    
- 한 case에 1-turn만 유지
    
- public/private는 절대 섞지 않는다

## 10. 오프라인 eval 설계: 이거 없으면 절대 출시하지 말 것

초기 gold set은 **500건**으로 시작한다.  
4개 축으로 고르게 뽑는다.

- public / private
    
- 팀 / 실 / 본부
    
- reviewer seen / unseen
    
- easy / hard case
    

평가 지표는 이 6개만 본다.

- **Acceptance rate**
    
- **Normalized edit distance**
    
- **Required term coverage**
    
- **Leakage rate**
    
- **Grounding rate**
    
- **Latency p95**
    

출시 기준은 이렇게 잡겠다.

- leakage rate = **0**
    
- required term coverage ≥ **95%**
    
- grounding rate ≥ **90%**
    
- acceptance rate ≥ **40%** on pilot
    
- p95 latency ≤ **6초** (hard case ≤ 9초)
    

OpenAI는 eval workflow를 **run and compare → continuous eval**로 운영하고, build 중에 datasets/evals를 계속 키우라고 권장한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/evaluation-best-practices/?utm_source=chatgpt.com "Evaluation best practices | OpenAI API"))

## 13. OpenAI 비용 추정

아래 가정으로 계산한다.

- writer 입력 6,500 tokens
    
    - 이 중 1,500 tokens는 캐시 가능한 고정 prefix
        
- writer 출력 500 tokens
    
- 전체 요청의 15%만 planner 추가 호출
    
- planner 입력 2,500 tokens / 출력 300 tokens
    

현재 공개 가격 기준으로 `gpt-4.1-mini`는 input $0.40 / cached input $0.10 / output $1.60, `o4-mini`는 input $1.10 / cached input $0.275 / output $4.40 per 1M tokens다. `text-embedding-3-small`은 $0.02 / 1M tokens다. Fine-tuning training은 `gpt-4.1-mini`가 $5.00 / 1M training tokens다. ([OpenAI Developers](https://developers.openai.com/api/docs/pricing/ "https://developers.openai.com/api/docs/pricing/"))

이 가정이면 **운영 평균 요청 단가**는 대략 **$0.0036 / case** 수준이다.  
따라서 월 처리량 기준 예산은 대략 이렇다.

- 월 1만 case: **약 $36**
    
- 월 5만 case: **약 $178**
    
- 월 10만 case: **약 $356**
    

즉, 이 시스템에서 모델 추론비는 보통 **주요 비용 항목이 아니다**. 대개 더 큰 비용은 인건비와 운영 복잡도다.

### 임베딩 백필 비용

300,000건 × 평균 250토큰이면 75M tokens다.  
`text-embedding-3-small` 기준 약 **$1.5** 정도다.

### SFT 1차 비용

30,000 예제 × 평균 1,200 training tokens로 잡으면 약 36M tokens다.  
`gpt-4.1-mini` training 단가로 **약 $180** 수준이다.

### DPO 1차 비용

10,000 pair × 평균 1,500 training tokens면 약 15M tokens다.  
대략 **약 $75** 수준이다.

요점은 명확하다.  
**fine-tuning 비용은 생각보다 작고, 데이터 정제와 운영 설계 비용이 훨씬 크다.**

## 17. 내 최종 권고

이 프로젝트는 **작은 데모**가 아니라 **중형 제품 개발**로 봐야 한다.  
그래서 접근 순서는 반드시 이렇게 가야 한다.

**1단계**: snapshot + retrieval + structured output + validator + human approval  
**2단계**: SFT로 회사 포맷/톤 안정화  
**3단계**: 운영 로그가 쌓이면 DPO로 선호 학습  
**4단계**: 비용/지연 최적화

핵심은 이거다.

**회사용어/조직맥락은 retrieval로, 형식/문체는 SFT로, 선호는 DPO로, 사고방지는 validator와 human approval로 푼다.**

원하면 다음 답변에서 바로 이어서  
**DB 스키마(SQL 초안) + FastAPI 엔드포인트 명세 + Structured Output JSON Schema + SFT/DPO용 JSONL 샘플**까지 바로 써주겠다.
