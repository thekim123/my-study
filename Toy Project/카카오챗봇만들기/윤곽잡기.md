이 경우는 **“LLM 하나 파인튜닝해서 해결”**로 보면 거의 틀어진다.  
정확히는 **평가 데이터 조인 시스템 + 검색 시스템 + 스타일 학습 + 검증기**를 묶은 **의사결정 보조형 작성 시스템**으로 봐야 한다.

내 권장은 한 줄로 요약하면 이렇다.

**`정규화된 evaluation case snapshot` + `다층 retrieval` + `SFT` + `DPO` + `validator` + `human approval`**

이 조합이 맞는 이유는 분명하다. SFT는 원하는 **스타일/형식/콘텐츠**를 더 안정적으로 내도록 만드는 데 적합하고, DPO는 **주관적 선호**가 개입되는 톤·문체·선호 표현을 맞추는 데 적합하다. 반면 회사 용어, 조직 목표, 최신 정책처럼 **동적으로 바뀌거나 케이스별로 달라지는 지식**은 retrieval/vector store로 넣는 편이 안전하다. 그리고 프로덕션에서는 출력 형식을 JSON schema로 강제하고, eval을 지속적으로 돌려야 운영이 된다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "Supervised fine-tuning | OpenAI API"))

## 먼저, 요구사항을 컴포넌트로 분해해야 한다

당신이 말한 요구사항은 사실 서로 다른 문제다.

- **관계가 복잡하다** → 원천 데이터를 LLM에 던지면 안 되고, 먼저 **케이스 단위 스냅샷**으로 정리해야 한다.
    
- **회사 용어를 꼭 써야 한다** → 용어집/가이드/과거 우수 코멘트를 **검색으로 주입**하고, 후단에서 **용어 사용 검증**을 해야 한다.
    
- **작성자 어투를 반영해야 한다** → 개인/팀 스타일 프로필을 만들고, 그걸 retrieval과 preference 학습으로 반영해야 한다.
    
- **공개/비공개가 섞여 있다** → 프롬프트 수준이 아니라 **데이터 접근 경로 자체를 분리**해야 한다.
    
- **프로덕션이다** → structured output, continuous eval, caching, security, HITL가 기본이다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/structured-outputs/ "Structured model outputs | OpenAI API"))
    

## 1) 핵심은 raw relation이 아니라 `evaluation_case_snapshot`이다

관계가 복잡하다고 해서 모델이 DB 조인을 직접 하게 만들면 안 된다.  
백엔드에서 미리 한 건의 2차평가 추천에 필요한 정보를 **정규화된 JSON 한 덩어리**로 만들어야 한다.

예를 들면 이런 형태다.

```json
{
  "cycle": "2026-H1",
  "employee": {
    "role_family": "backend",
    "level": "senior"
  },
  "org_context": {
    "team": "...",
    "group": "...",
    "division": "..."
  },
  "goal_tree": [
    {"level": "division", "goal": "..."},
    {"level": "team", "goal": "..."},
    {"level": "individual", "goal": "..."}
  ],
  "missions": [
    {
      "mission_id": "...",
      "summary": "...",
      "peer_scores": {"ownership": 4.2, "collaboration": 3.8},
      "peer_comment_summary": "...",
      "manager_scores": {"impact": 3.5, "execution": 4.0},
      "manager_public_comment": "...",
      "manager_private_comment": "..."
    }
  ],
  "review_scope": "private",
  "second_reviewer_id": "...",
  "policy_profile": {
    "must_use_terms": ["고객가치", "오너십"],
    "forbidden_patterns": ["단정적 인신평가", "근거 없는 추정"]
  },
  "style_profile_ref": "reviewer_or_team_style_profile"
}
```

여기서 중요한 점은 두 가지다.  
첫째, **점수 집계·평균·편차·조직 트리 펼치기 같은 계산은 전부 코드로 처리**해야 한다. 이런 것은 LLM에게 맡길 일이 아니다. 둘째, **public snapshot**과 **private snapshot**을 별도로 만들어야 한다. public 추천 경로에서는 private 필드가 애초에 로드되지 않아야 한다. 프로덕션 최적화 관점에서도 입력 토큰과 요청 수를 줄이고, “LLM을 기본값으로 쓰지 말라”는 원칙이 유리하다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/latency-optimization/ "Latency optimization | OpenAI API"))

내 의견으로는, 이 단계에서 **GraphRAG부터 시작할 필요는 없다**. 우선은 RDB/warehouse에서 케이스 스냅샷을 materialize하고, 그래프 탐색이 진짜 필요한 경우에만 뒤에 붙이는 편이 안전하다.

## 2) 검색 계층은 하나가 아니라 최소 3개다

이 시스템에서 retrieval은 “문서 검색”이 아니라 **컨텍스트 주입 레이어**다.  
나는 최소 아래 3개를 분리하겠다.

### A. 유사 평가 사례 검색

과거 2차평가 final comment를 대상으로, 다음 조건을 섞어서 찾는다.
- 역할군/직무 family
- 비슷한 score pattern
- 비슷한 peer-manager 간 신호
- 같은 평가 차원(예: 협업, 실행, 리더십)
- 같은 공개/비공개 scope
    
여기서는 vector search가 유효하다. 임베딩은 relatedness를 측정하고 search/recommendation에 흔히 쓰인다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/embeddings/ "Vector embeddings | OpenAI API"))

### B. 회사 용어/정책/정신 검색

여기는 별도 인덱스가 필요하다.

- 회사 핵심가치 정의
    
- 평가 가이드라인
    
- 금지 표현/권장 표현
    
- 조직별 목표 설명
    
- 우수 코멘트 예문 bank
    

이건 exact jargon이 중요하므로 **semantic + keyword hybrid**가 맞다. OpenAI의 file search도 semantic과 keyword search를 함께 사용하도록 설계되어 있다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/tools-file-search/ "File search | OpenAI API"))

### C. 작성자 스타일 검색

이건 개인화의 핵심이다.

- 해당 2차평가자 본인의 과거 승인 코멘트
    
- 데이터가 부족하면 팀장군/조직군 스타일
    
- 문체 축: 직설적/코칭형/간결/완곡/근거우선
    
- 자주 쓰는 연결어, 마무리 표현, 칭찬/보완 비율
    

다만 이건 **무조건 개인 스타일을 복제**하면 안 된다.  
항상 **회사 정책 > 조직 표준 > 개인 스타일** 순서여야 한다.  
즉, 개인 어투는 허용하되 회사 기준을 깨면 안 된다.

## 3) 파인튜닝은 하나가 아니라 2단으로 보는 게 좋다

### 1단: SFT

SFT의 목표는 **사실 기억**이 아니라 **출력 계약(output contract)**을 고정하는 것이다.

학습시키는 것:

- 공개/비공개별 문장 구조
    
- 회사가 선호하는 표현 방식
    
- 근거 기반 서술 패턴
    
- 평가 코멘트의 길이/톤/포맷
    
- “강점 + 개선 포인트 + 기대 행동” 같은 회사 고유 템플릿
    

SFT는 원하는 스타일과 콘텐츠를 더 안정적으로 만들고, 더 짧은 프롬프트로도 같은 동작을 내게 해서 비용/지연을 낮추는 데 도움이 된다. 그리고 학습 데이터는 실제 운영 입력과 최대한 비슷해야 한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/model-optimization/ "Model optimization | OpenAI API"))

여기서 아주 중요하다.  
**SFT 입력은 raw DB dump가 아니라 실제 inference 때 쓰는 snapshot 형식과 유사해야 한다.**  
실서비스에서 snapshot + retrieved context로 생성할 거면, 학습 데이터도 그 구조를 닮아야 한다. 그래야 inference-train mismatch가 줄어든다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/supervised-fine-tuning/ "Supervised fine-tuning | OpenAI API"))

### 2단: DPO

DPO는 이 시스템에 특히 잘 맞는다. 이유는 코멘트 추천은 상당 부분 **주관적 선호 문제**이기 때문이다.

예를 들어 같은 케이스에 대해

- A안: 너무 딱딱하지만 안전한 문장
    
- B안: 더 자연스럽고 회사 어투에 맞는 문장
    

이런 두 개가 있을 때, 어떤 쪽을 더 선호하는지 학습시키는 데 DPO가 맞다. 공식 문서도 DPO를 **subjective decision-making**, **tone and style**에 적합한 방식으로 설명한다. 데이터 포맷도 `prompt + preferred + non-preferred`다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/direct-preference-optimization/ "Direct preference optimization | OpenAI API"))

당신 케이스에서 DPO 데이터는 이렇게 만들 수 있다.

- prompt: evaluation_case_snapshot + retrieved context
    
- preferred: 최종 승인된 2차 코멘트
    
- non-preferred: 초기 추천안 / rejected 후보 / 수정 전 초안
    

즉, **사람이 어떻게 고쳤는지**가 가장 중요한 학습 신호가 된다.

## 4) “회사 용어를 꼭 써야 한다”는 요구는 FT만으로 풀지 않는다

이 요구를 파인튜닝 하나로 해결하려 하면 유지보수 지옥이 된다.

권장 방식은 이렇다.

1. **용어집/가치체계/권장 표현집**을 별도 knowledge base로 관리
    
2. 요청마다 관련 용어를 retrieval로 주입
    
3. 생성 후 validator가
    
    - 필수 용어 포함 여부
        
    - 금지 표현 포함 여부
        
    - scope 위반 여부
        
    - 근거 없는 판단 여부  
        를 검사
        
4. 미통과면 재생성
    

즉, **용어는 retrieval + validation**, **문체는 SFT/DPO**로 분리해야 한다. retrieval은 knowledge base를 모델에 연결해 동적 지식을 주입하는 용도고, structured outputs는 이런 검증 흐름을 안정적으로 구현하는 데 유리하다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/tools-file-search/ "File search | OpenAI API"))

회사 “정신”도 추상어로 두면 안 된다.  
반드시 기계가 다룰 수 있게 바꿔야 한다.

예를 들면:

- `ownership`
    
- `collaboration`
    
- `customer_value`
    
- `execution_excellence`
    

이런 식으로 **정의 + 대표 표현 + 금지 반례**를 붙인 taxonomy로 만들어야 한다.  
그 후 각 코멘트가 어떤 value tag를 반영했는지 기록하게 하면 된다.

## 5) “평소 어투 반영”은 개인 파인튜닝보다 `style profile + fallback hierarchy`가 안전하다

개인 어투를 바로 FT에 박아 넣는 접근은 과적합 위험이 크다.  
프로덕션에서는 보통 다음 우선순위를 둔다.

1. reviewer 개인 스타일
    
2. reviewer가 속한 조직/직군 스타일
    
3. 회사 기본 스타일
    

개인 데이터가 충분할 때만 1번을 쓰고, 부족하면 2번 또는 3번으로 내려간다.  
이걸 위해 각 reviewer에 대해 다음 같은 **style profile**을 만든다.

- 평균 문장 길이
    
- 존댓말/완곡어 사용 정도
    
- “강점 먼저, 보완 나중” 비율
    
- 자주 쓰는 회사 표현
    
- 금지해야 할 편향/과도한 직설 패턴
    

실시간 생성 시에는 과거 코멘트 원문을 통째로 많이 넣기보다,  
**대표 예문 2~3개 + 스타일 요약 feature**를 넣는 쪽이 낫다.  
이 방식은 토큰과 지연을 줄이기 쉽고, prompt caching에도 유리하다. cache hit를 늘리려면 고정 지시문과 예시는 앞에, 케이스별 데이터는 뒤에 배치하는 것이 권장된다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/prompt-caching/ "Prompt caching | OpenAI API"))

## 6) 서빙 플로우는 “생성기”가 아니라 “오케스트레이션”으로 짜야 한다

내가 실제로 만들면 요청 하나를 이렇게 처리한다.

1. **Case Snapshot Service**  
    원천 HR 테이블을 조인해서 `evaluation_case_snapshot` 생성
    
2. **Policy/ACL Filter**  
    public 요청이면 private 필드 제거  
    reviewer 권한에 맞지 않는 자료 제거
    
3. **Retrieval**
    
    - 유사 사례 top-k
        
    - 회사 용어/정책 top-k
        
    - 스타일 exemplar top-k
        
4. **Planner 단계**  
    케이스가 복잡하면 reasoning-capable model이
    
    - 핵심 강점
        
    - 우려 포인트
        
    - 필요한 company terms
        
    - 공개/비공개 구분  
        을 먼저 정리
        
5. **Writer 단계**  
    lower-latency GPT 계열이 실제 추천 문안 2~3개 생성
    
6. **Validator 단계**
    
    - JSON schema 통과 여부
        
    - 필수 용어 포함 여부
        
    - 금지 표현 여부
        
    - public/private leakage 여부
        
    - 근거 span 존재 여부
        
    - 점수와 코멘트의 정합성
        
7. **UI 노출**  
    추천안 + 사용 근거 + risk flag를 함께 표시
    
8. **Human approval**  
    선택/수정/거절 로그 저장
    

이 planner/doer 분리는 현재 OpenAI 공식 가이드의 “reasoning model은 planner, GPT model은 workhorse” 패턴과 맞닿아 있다. 복잡한 다단 문제는 reasoning 계열이 전략/판단을 맡고, 빠른 실행은 GPT 계열이 맡는 구성이 권장된다. 툴 호출과 structured outputs를 같이 쓰면 이런 워크플로우를 훨씬 안정적으로 짤 수 있다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/reasoning-best-practices/ "Reasoning best practices | OpenAI API"))

## 7) 출력은 자유문장이 아니라 구조화된 객체여야 한다

프로덕션에서는 최종 응답을 바로 문장 하나로 받지 말고, 먼저 이렇게 받는 편이 낫다.

```json
{
  "scope": "private",
  "drafts": [
    {
      "text": "...",
      "tone": "coaching",
      "required_terms_used": ["오너십", "협업"],
      "evidence_refs": ["peer:collaboration", "manager:execution"],
      "risk_flags": []
    }
  ],
  "summary_of_reasoning": {
    "strengths": ["..."],
    "improvement_points": ["..."]
  }
}
```

Structured Outputs는 모델 응답을 개발자가 정의한 JSON schema에 맞추도록 보장하는 기능이라, 키 누락이나 enum 오류 같은 문제를 줄이는 데 적합하다. 이런 구조를 쓰면 validator, UI, audit log를 붙이기가 쉬워진다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/structured-outputs/ "Structured model outputs | OpenAI API"))

## 8) 평가와 운영 없이는 절대 프로덕션이 안 된다

공식 eval 가이드도 먼저 **목표 정의 → 데이터셋 수집 → 지표 정의 → 비교 실험 → continuous eval** 순서를 권장한다. 또 LLM은 open-ended generation보다 **pairwise comparison / scoring / classification** 쪽이 더 신뢰도 높은 평가가 되기 쉽다고 설명한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/evaluation-best-practices/ "Evaluation best practices | OpenAI API"))

당신 시스템에서는 최소 아래 지표를 봐야 한다.

- **Acceptance rate**: 추천안을 거의 수정 없이 채택한 비율
    
- **Edit distance**: 추천안과 최종 저장 코멘트 차이
    
- **Term coverage**: 필수 도메인 용어 반영률
    
- **Leakage rate**: public 초안에 private 정보가 섞인 비율
    
- **Grounding rate**: 실제 입력 근거가 없는 문장이 포함된 비율
    
- **Human preference**: HR/평가자 pairwise 선호도
    
- **Latency / cost per request**
    

그리고 테스트셋은 반드시 분리해야 한다.

- 시간 분리: 지난 반기 → 다음 반기
    
- 조직 분리: 본 적 없는 팀/실/본부
    
- reviewer 분리: 본 적 없는 2차평가자
    
- scope 분리: public/private
    
- hard case 분리: peer와 manager 신호가 충돌하는 케이스
    

## 9) 안전장치는 HR 도메인이라 더 강해야 한다

이건 high-stakes에 가깝다.  
따라서 기본값은 **추천 시스템**이지 **자동 확정 시스템**이 아니다.

OpenAI safety best practices도 high-stakes 영역에서는 human review를 권장하고, adversarial testing/red-teaming을 하라고 명시한다. 또한 production best practices는 보안·컴플라이언스·익명화·비밀 관리 등을 점검하라고 한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/safety-best-practices/ "Safety best practices | OpenAI API"))

실제로는 아래를 필수로 넣는 편이 낫다.

- public/private 완전 분리 경로
    
- reviewer 권한 기반 retrieval filter
    
- 승인 전 human review
    
- red-team 시나리오
    
    - private 문장을 public에 섞기
        
    - 근거 없는 부정 평가 생성
        
    - 특정 개인에 대한 과도한 단정
        
    - 금지 용어 우회
        
- 서버사이드 키 관리와 감사 로그
    
- 가능한 범위의 비식별화/암호화
    

## 10) 비용/지연은 나중에 줄이면 된다. 다만 구조는 처음부터 그렇게 깔아라

처음부터 가장 싼 모델로 버티기보다, 먼저 정확도를 맞추고 나중에 작은 모델로 내리는 것이 일반적으로 낫다. 공식 model selection 가이드도 정확도를 먼저 맞춘 뒤, 더 작은 모델 zero/few-shot 비교나 distillation/fine-tuning으로 비용과 지연을 줄이는 흐름을 제시한다. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/model-selection/ "Model selection | OpenAI API"))

그래서 현실적인 순서는 보통 이렇다.

1. **RAG + structured output + validator**로 baseline
    
2. 그 위에 **SFT**로 회사 포맷/톤 고정
    
3. 운영 로그가 쌓이면 **DPO**로 사람 선호 반영
    
4. 품질이 고정되면 **smaller model/distillation**로 비용 최적화
    
5. 캐시가 잘 먹도록 고정 프롬프트를 앞에 두고 dynamic context는 뒤로 배치 ([OpenAI Developers](https://developers.openai.com/api/docs/guides/prompt-caching/ "Prompt caching | OpenAI API"))
    

## 내 결론

당신 상황에서 정답은 **“파인튜닝을 하되, 파인튜닝만 믿지 말라”**다.

정확히는 이렇게 가는 게 맞다.

- **사실/관계/조직목표/회사용어**는 retrieval
    
- **형식/톤/문장 패턴**은 SFT
    
- **사람이 더 선호하는 표현 선택**은 DPO
    
- **공개/비공개 안전성**은 ACL + validator
    
- **최종 책임**은 human approval
    

즉, 프로덕션 아키텍처의 중심은 **모델**이 아니라 **데이터 계약과 검증 파이프라인**이다.

다음 단계로는, 이걸 바로 구현 가능한 수준으로 내려서  
**DB/인덱스 스키마**, **프롬프트 계약**, **SFT/DPO 학습셋 포맷**, **오프라인 평가셋 설계**까지 끊어서 잡는 게 맞다.
