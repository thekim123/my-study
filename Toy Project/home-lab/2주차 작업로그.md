## 2주차: “RAG 앱”에서 “에이전트”로 올리기
이번 주 목표는 **도구 선택과 라우팅**이다.

### 할 일
1. **도구를 2개만 둔다**
    - `retrieve_docs`
    - `search_repo` 또는 `grep_repo`
        
2. **에이전트 라우팅 추가**
    - 일반 설명 질문이면 retrieval
    - 코드 위치 찾기면 repo search
    - 아주 단순한 질문이면 바로 답변
    - 애매하면 먼저 retrieval
        
3. **도구 호출 결과를 응답에 반영**
    - “어떤 도구를 썼는지”
    - “왜 그 답이 나왔는지”
    - “어떤 파일을 봤는지”
        
4. **실패 케이스 수집**
    - retrieval은 했는데 엉뚱한 chunk를 고른 경우
    - repo search는 맞았는데 최종 요약이 틀린 경우
    - 답은 맞지만 출처가 부정확한 경우

### 주말까지 나와야 하는 결과
```text
agent("토큰 갱신 로직은 어디에 있어?")
-> repo search 사용
-> 경로 제시
-> 코드 설명
```

### 이번 주 핵심 판단
- **LangChain으로 충분하면 그대로 간다**
- 노드 분기, 상태, 재시도, human-in-the-loop가 필요할 때만 **LangGraph**로 내려간다
    
즉, 2주차의 목적은 **LangGraph를 쓰는 것**이 아니라  
**정말 필요할 때만 쓰게 되는 상태를 만드는 것**이다.

##  ✅1. main.py 기준으로 에이전트 API 스펙부터 정해.
- POST /agent를 만들고 입력은 question, 출력은 used_tool, reason, sources, answer를 항상 반환하도록 고정
##  ✅2. retrieve_docs부터 먼저 구현(1순위).
- loader.load_vector_store, loader.search_chunks를 감싸서 retrieve_docs(question)를 만들고, 반환값에 chunk_id/source_path/start/end/score를 붙여줘.
##  3. search_repo(또는 grep_repo) 구현(2순위).
- 지금 코드베이스 기준으로 project_root 안에서 키워드 포함 라인을 찾아 path, line, snippet 형태로 반환하게 해.
- (처음엔 search_repo로 통일하고, 나중에 grep_repo 별도 별칭으로 분기해도 됨)
##  ✅4. 라우팅 로직 추가(가장 먼저):
- “어디에 있지?” / “어떤 파일” / “어디에 작성” 류면 search_repo
- “무슨 의미지?”, “설명해줘”, “요약해줘” 류면 retrieve_docs
- 인사/날씨/짧은 대화형은 바로 답변
- 판정 못 하면 retrieve_docs 기본값
##  ✅5. 응답 템플릿을 강제.
   항상 아래 항목을 채워서 반환:
	- used_tool
	- route_reason (왜 그 도구를 골랐는지)
	- source_files (최소 1개 이상)
	- answer + confidence(0~1)
##  ✅6. 실패 케이스 수집을 먼저 적어 테스트로 박아.
- retrieval이 엉뚱한 chunk 반환
- repo search는 맞았는데 답변 요약이 틀림
- 정답은 맞는데 파일/범위가 부정확함
- 각 케이스 당 “원인 라벨 + 개선안” 붙여서 로그 남기기.
