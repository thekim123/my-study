좋다. 이번 4주는 **“무언가를 많이 배우는 기간”이 아니라 “하나를 끝내는 기간”**으로 잡자.

현재 공식 문서 흐름 기준으로는, 새 프로젝트는 **OpenAI Responses API**를 중심으로 두고, 에이전트 시작은 **LangChain**, 더 세밀한 라우팅이나 상태 관리가 필요할 때만 **LangGraph**로 내려가며, 디버깅과 평가는 **LangSmith**, 배포 실험장은 **Proxmox 기반 홈랩**으로 잡는 구성이 가장 정합적이다. OpenAI는 새 텍스트 생성 프로젝트에 Responses API를 권장하고 있고, Assistants API는 **2026년 8월 26일 종료 예정**이라고 안내한다. LangChain은 빠른 시작용 pre-built agent architecture를 제공하고, LangGraph는 durable execution·streaming·human-in-the-loop 같은 낮은 수준의 orchestration에 초점을 둔다. LangSmith는 tracing/evaluation workflow를 제공하고, Proxmox VE는 KVM과 LXC를 한 플랫폼에서 관리한다. ([OpenAI 개발자 센터](https://developers.openai.com/api/docs/guides/text/?utm_source=chatgpt.com "Text generation | OpenAI API"))

## 이번 4주의 단일 목표

**프로젝트 하나를 끝낸다.**

주제는 이걸 추천한다.

**“내 문서/노트/코드 리포를 질의할 수 있는 개발자용 지식 에이전트”**

최종 상태는 이렇다.
- 네 노트 폴더 + 리포 1개를 읽을 수 있다
- 질문하면 답을 한다
- 근거가 된 문서 조각이나 파일 경로를 같이 준다
- 필요하면 검색 툴을 호출한다
- trace가 남는다
- 20개 정도의 평가 질문셋으로 성능을 본다
- 홈랩에 배포되어 실제로 접근 가능하다

## 1주차: 범위 고정 + 로컬 MVP
이번 주 목표는 **“일단 돌아가는 것”**이다.

### 할 일

1. **범위를 자른다**
    - 리포는 1개만
    - 문서 폴더도 1개만
    - PDF까지 한꺼번에 넣지 말고, 우선 markdown/text/code 위주로 시작
    - UI도 웹 말고 CLI 또는 아주 얇은 FastAPI 하나면 충분
        
2. **기본 파이프라인 만든다**
    
    - 문서 로드
        
    - chunking
        
    - 임베딩/인덱싱
        
    - 질문 → retrieval → 답변 생성
        
    - 답변에 source path 붙이기
        
3. **답변 정책 정한다**
    
    - 모르면 모른다고 말하기
        
    - 근거 없으면 추측하지 않기
        
    - 답변 끝에 참고한 파일 2~5개 보여주기
        
4. **실행 로그 남기기**
    
    - 질문
        
    - 검색된 문서 수
        
    - 선택된 chunk
        
    - 최종 응답
        

### 주말까지 나와야 하는 결과

```text
ask("이 프로젝트에서 인증 흐름이 어떻게 되어 있지?")
-> 답변
-> 참고 파일: auth/service.ts, docs/login-flow.md
```

### 이번 주 체크포인트

- 30~50개 문서 수준에서 돌아가는가
    
- 근거 파일이 실제로 맞는가
    
- 헛소리 비율이 높은가 낮은가
    

### 이번 주 금지

- 멀티에이전트
    
- 예쁜 프론트
    
- 복잡한 DB
    
- 홈랩 설정놀이

## 2주차: “RAG 앱”에서 “에이전트”로 올리기

이번 주 목표는 **도구 선택과 라우팅**이다.

### 할 일

1. **도구를 2개만 둔다**
    
    - `retrieve_docs`
        
    - `search_repo` 또는 `grep_repo`
        
2. **에이전트 라우팅 추가**
    
    - 일반 설명 질문이면 retrieval
        
    - 코드 위치 찾기면 repo search
        
    - 아주 단순한 질문이면 바로 답변
        
    - 애매하면 먼저 retrieval
        
3. **도구 호출 결과를 응답에 반영**
    
    - “어떤 도구를 썼는지”
        
    - “왜 그 답이 나왔는지”
        
    - “어떤 파일을 봤는지”
        
4. **실패 케이스 수집**
    
    - retrieval은 했는데 엉뚱한 chunk를 고른 경우
        
    - repo search는 맞았는데 최종 요약이 틀린 경우
        
    - 답은 맞지만 출처가 부정확한 경우
        

### 주말까지 나와야 하는 결과

```text
agent("토큰 갱신 로직은 어디에 있어?")
-> repo search 사용
-> 경로 제시
-> 코드 설명
```

### 이번 주 핵심 판단

- **LangChain으로 충분하면 그대로 간다**
    
- 노드 분기, 상태, 재시도, human-in-the-loop가 필요할 때만 **LangGraph**로 내려간다
    

즉, 2주차의 목적은 **LangGraph를 쓰는 것**이 아니라  
**정말 필요할 때만 쓰게 되는 상태를 만드는 것**이다.

## 3주차: tracing + eval + 품질 개선

이번 주 목표는 **“대충 잘 되는 느낌”을 버리고, 어디서 망하는지 보이게 만드는 것”**이다.

LangSmith 쪽 공식 문서도 tracing에서 retrieval step과 LLM call을 함께 보고, evaluation에서는 dataset·target function·evaluators 구조로 점검하는 흐름을 제시한다. 이번 주는 그걸 그대로 네 프로젝트에 적용하면 된다. ([LangChain Docs](https://docs.langchain.com/langsmith/observability-quickstart?utm_source=chatgpt.com "Tracing quickstart - Docs by LangChain"))

### 할 일

1. **LangSmith tracing 붙이기**
    
    - 질문 1회마다 trace 1개
        
    - 어떤 도구를 썼는지
        
    - 어떤 chunk를 참고했는지
        
    - 응답까지 얼마나 걸렸는지
        
2. **평가셋 20개 만든다**
    
    - 쉬운 질문 5개
        
    - 파일 위치 찾기 5개
        
    - 여러 문서를 합쳐야 하는 질문 5개
        
    - 답이 없어야 하는 질문 5개
        
3. **실패 유형 분류**
    
    - retrieval miss
        
    - chunking 문제
        
    - tool selection 문제
        
    - prompt 문제
        
    - answer formatting 문제
        
4. **한 번에 하나씩 고친다**
    
    - chunk size 바꾸기
        
    - top-k 조정
        
    - rerank 여부 점검
        
    - system prompt 정리
        
    - “모르면 모른다” 기준 강화
        

### 주말까지 나와야 하는 결과

- 평가 질문셋 20개
    
- baseline 결과
    
- 개선 후 결과
    
- “뭐가 문제였고 어떻게 고쳤는지” 한 페이지 정리
    

### 이번 주의 진짜 산출물

코드보다 더 중요한 게 이거다.

**“내 시스템은 주로 어디서 틀리는가”를 내가 설명할 수 있게 되는 것**

이게 생기면 그냥 장난감이 아니라 포트폴리오가 된다.

## 4주차: 홈랩 배포 + 문서화 + 데모화

이번 주 목표는 **“내 컴퓨터에서만 되던 것”을 끝내는 것**이다.

### 할 일

1. **배포 형태 최소화**
    
    - Docker 하나 또는 compose 하나
        
    - FastAPI + 간단한 web/CLI
        
    - 환경변수 분리
        
    - 로그 파일 분리
        
2. **홈랩 반영**
    
    - Proxmox 위에 VM 또는 LXC 하나
        
    - 거기에 앱 올리기
        
    - 내부망에서 접속 확인
        
    - 재부팅 후 자동 복구 확인
        
3. **운영 체크**
    
    - 인덱스 재생성 방법 문서화
        
    - 데이터 폴더 백업 경로 정리
        
    - API key 관리 방식 정리
        
    - 장애났을 때 재시작 절차 정리
        
4. **포트폴리오화**
    
    - README 정리
        
    - 아키텍처 그림 1장
        
    - 데모 GIF 또는 짧은 영상
        
    - “왜 이런 구조를 선택했는지” 작성
        

### 주말까지 나와야 하는 결과

- 홈랩에서 접근 가능한 실행본
    
- README
    
- 데모 자료
    
- 회고 문서
    

### 여기서 중요한 것

홈랩은 **앱을 띄우기 위한 수단**이어야지,  
**클러스터/HA/Ceph 놀이**로 가면 안 된다.

Proxmox 자체는 KVM과 LXC를 한 인터페이스로 다루기 좋기 때문에 홈랩 베이스로는 충분하다. 하지만 이번 4주의 목적은 인프라 연구가 아니라 **배포 완료**다. ([Proxmox](https://www.proxmox.com/en/products/proxmox-virtual-environment/overview?utm_source=chatgpt.com "Proxmox Virtual Environment"))

## 알고리즘은 이렇게만 가져가라

알고리즘은 지금 주력 트랙이 아니라 **보조 트랙**이다.

### 1주차

- trie
    
- inverted index 개념
    
- 문자열 탐색 감각
    

### 2주차

- B-tree / B+ tree
    
- 왜 DB 인덱스가 그렇게 생겼는지 이해
    

### 3주차

- heap / priority queue
    
- LRU cache
    

### 4주차

- bloom filter
    
- consistent hashing 또는 union-find 중 하나
    

포인트는 **강의 완강**이 아니다.  
매주 1개만 잡고,  
`개념 이해 → 손으로 구현 → 어디에 쓰이는지 연결`  
이 3개만 하면 된다.
