• 이 프로젝트는 이미 얇은 RAG 골격이 있습니다. 현재 기준점은 이렇습니다.

  - app/loader.py 가 vector_store.json을 읽어 메모리에서 cosine search를 합니다.
  - app/chunker.py 는 문자 수 기준 청킹만 하고, 코드 구조나 줄 번호 정확도는 약합니다.
  - app/embedder.py 는 Bedrock 임베딩과 생성 호출을 한 클래스에 같이 넣어둔 상태입니다.
  - app/agent/service.py 는 search_repo와 retrieve_docs를 라우팅하고, app/main.py 는 /agent 하나만 노출합니다.

  즉, 방향은 새 시스템을 갈아엎는 게 아니라, 로컬 JSON 벡터 저장을 OpenSearch 인덱스로 바꾸고, 정적 코드 검색을 Git diff + Bedrock 작업까지 확장하는 쪽이 맞습니다.

  권장 설계

  1. 저장소 계층을 먼저 분리합니다. load_vector_store() 같은 파일 기반 접근을 직접 쓰지 말고, ChunkRepository 인터페이스 뒤에 JsonChunkRepository와 OpenSearchChunkRepository를 둡니다. 이렇게 해야 /agent 흐름을 거의 안 깨고 교체
     할 수 있습니다.
  2. 청크 스키마를 먼저 고정합니다. 최소 필드는 chunk_id, repo, branch, commit, file_path, language, start_line, end_line, content, content_hash, embedding 입니다. 지금처럼 start/end를 문자 오프셋으로만 두면 소스 참조 품질이 떨
     어집니다.
  3. 코드 청킹은 처음부터 AST에 집착하지 말고, 1차는 줄 번호 기반 윈도우 + 함수/클래스 헤더 휴리스틱으로 갑니다. 이 프로젝트가 여러 언어를 다룰 가능성이 있어서, 초기엔 범용 전략이 낫습니다. 현재 app/chunker.py 는 fallback 로직으
     로만 남기는 게 좋습니다.
  4. OpenSearch 인덱스는 두 개로 나눕니다. code_chunks 와 git_diff_chunks 입니다. 코드 청크와 diff hunk를 한 인덱스에 섞으면 필터링과 프롬프트 조립이 지저분해집니다.
  5. 검색 경로는 하이브리드로 갑니다. 벡터 검색만 쓰지 말고 file_path, symbol, keyword 조건을 같이 태웁니다. 현재 app/agent/service.py 의 search_repo() 문자열 매칭은 여기에 흡수되면 됩니다.
  6. Bedrock 역할을 둘로 분리합니다. 임베딩용 클라이언트와 생성용 클라이언트를 분리하고, region/model/env 도 분리합니다. 지금 app/embedder.py 는 책임이 섞여 있어서 운영 시점에 바로 불편해집니다.
  7. Git diff 인덱싱은 별도 파이프라인으로 만듭니다. git diff base..head 나 git show commit 에서 hunk 단위로 끊고, 각 hunk를 임베딩해서 git_diff_chunks 에 넣습니다. 질의가 “최근 뭐 바뀌었어”, “이 변경 리뷰해줘” 류면 diff 인덱스
     를 먼저 보고, 필요하면 관련 코드 청크를 2차로 가져옵니다.

  구현 단계

  8. 환경설정 정리
      - app/config.py 추가.
      - TARGET_REPO_ROOT, OPENSEARCH_HOST, OPENSEARCH_INDEX_CODE, OPENSEARCH_INDEX_DIFF, BEDROCK_EMBED_MODEL_ID, BEDROCK_CHAT_MODEL_ID 분리.
      - 현재 app/embedder.py 의 하드코딩 region 도 같이 정리.
  9. OpenSearch 클라이언트/저장소 도입
      - app/search/opensearch_client.py
      - app/search/chunk_repository.py
      - 기존 app/loader.py 의 load_vector_store, search_chunks 역할을 여기로 이동.
  10. 색인 파이프라인 추가
      - app/indexing/file_loader.py
      - app/indexing/chunker.py
      - app/indexing/index_repo.py
      - 전체 인덱싱 1회 + content_hash 기반 증분 업데이트.
  11. 질의 경로 교체
      - app/agent/service.py 에서 in-memory chunks 대신 repository 검색 사용.
      - search_repo, retrieve_docs 를 OpenSearch 기반으로 재작성.
      - 응답 source 에 commit, end_line, source_type 추가 권장.
  12. Git diff 파이프라인 추가
      - app/git/diff_parser.py
      - app/indexing/index_diff.py
      - base/head 기준 diff 인덱싱과 commit 단건 인덱싱 둘 다 지원.
  13. Bedrock 작업 유형 확장
      - 일반 질의: 코드 청크 검색 후 답변
      - 변경 요약: diff 청크 기반 요약
      - 리뷰 요청: diff + 관련 코드 청크 같이 프롬프트에 넣고 위험요소/회귀 가능성 답변
  14. 운영용 엔드포인트
      - /agent 유지
      - /admin/index/full
      - /admin/index/diff
      - /admin/reindex/file
      - 처음엔 동기식으로 두고, 나중에 background job 으로 빼는 게 낫습니다.

  MVP 순서

  15. vector_store.json 의존 제거 후 OpenSearch에서 코드 청크 조회만 되게 만들기
  16. 전체 저장소 1회 인덱싱 CLI 만들기
  17. /agent 를 OpenSearch 검색 기반으로 전환
  18. Git diff 인덱싱 추가
  19. “변경 요약/리뷰”용 Bedrock 프롬프트 추가

  중요한 판단

  - 처음부터 tree-sitter 같은 정교한 파서를 넣기보다, 줄 번호 기반 청킹 + 증분 인덱싱을 먼저 성공시키는 게 맞습니다.
  - OpenSearch에는 벡터만 넣지 말고 원문, 해시, 커밋, 경로 메타데이터를 같이 넣어야 합니다.
  - Bedrock과 OpenSearch의 region/auth 정책은 초기에 맞춰야 합니다. 지금처럼 Bedrock region 이 코드에 박혀 있으면 나중에 바로 걸립니다.

  원하면 다음 단계로 바로 이어서, 이 프로젝트 기준으로 파일별 변경 계획이나 1차 구현 뼈대(app/config.py, OpenSearch repository, indexer CLI)까지 실제 코드 기준으로 쪼개드리겠습니다.
