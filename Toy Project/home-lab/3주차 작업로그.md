  ## 0) 전제/요구사항 정리

  1. OpenSearch: 벡터 검색 가능한 엔진(k-NN, 또는 AWS OpenSearch Service의 knn_vector)
  2. 임베딩: Bedrock 임베딩 모델(예: Titan Embed) 사용
  3. LLM: Bedrock 생성 모델(Claude/Qwen 등) 사용
  4. Git 연동: 파일 변경분(diff) 추적 및 커밋/브랜치 단위 메타데이터 저장
  5. 질의 타입:
      1. “이 코드 어디서 동작해?” (일반 RAG 질의)
      2. “최근 커밋에서 어떤 코드가 바뀌었나?”
      3. “변경사항 리뷰해줘 / 버그 가능성 점검해줘”

  ## 1) 전체 아키텍처

  - Ingestion 파이프라인
      - 저장소 스캔 → 코드 청크화 → 임베딩 생성 → OpenSearch upsert
  - Diff 파이프라인
      - git diff/git show → 변경 hunk 추출 → 별도 diff 인덱스에 저장
  - Retriever
      - 질의 벡터화 → kNN + keyword 검색 하이브리드
      - 결과 스코어 재정렬 + 필터(파일/브랜치/커밋)
  - RAG 오케스트레이터
      - 조회된 청크 + diff 컨텍스트를 프롬프트로 묶어 Bedrock 호출
  - 관리/운영
      - 인덱스 mapping 버전관리, 재인덱싱 전략, 데이터 수명주기

  ## 2) 인덱스 설계 (핵심)

  1. repo_code_chunks (기존 코드 청크)

  - 필수 메타: repo, branch, file_path, commit, chunk_id, start_line, end_line, language, updated_at
  - 검색 필드: content(text), embedding(knn_vector)

  2. repo_code_diffs (변경분)

  - 필수 메타: repo, branch, commit, parent_commit, file_path, change_type, hunk_id, updated_at
  - 검색 필드: diff_content, embedding

  3. 추천 매핑

  - 벡터 차원은 사용 임베딩 모델 차원과 정확히 일치
  - 텍스트는 text + 제한된 분석기
  - 필터는 keyword 기반으로 exact 매칭(파일 경로, commit, branch)

  ## 3) 파이프라인 상세 (실행 순서)

  1. 청크 전략 수립

  - 기본: 400~800 tokens, overlap 80~120
  - Python/JS는 AST/문법 기반 분절(함수·클래스 단위) 우선
  - 긴 텍스트만 고정 토큰 방식 fallback

  2. 전체 코드 초기 인덱싱

  - 브랜치별로 스캔
  - 동일 commit/파일은 content-hash로 upsert 판별
  - 파일 삭제/이동 시 이전 청크 삭제 로직 필요

  3. 임베딩 생성

  - 문서 배치 처리(동시성 제한)
  - 실패 재시도 정책(지수 backoff)
  - 벡터 캐시: 동일 chunk text 재임베딩 방지

  4. Git diff 인덱싱

  - pre-commit/CI/스케줄러에서 트리거
  - git diff <base>..<head> 기준으로 변경분 추출
  - 각 hunk에 대해 메타 포함해 별도 벡터 저장
  - 질문에서 “변경분 기준 질의” 시 repo_code_diffs 우선 검색 후 repo_code_chunks 보강 검색

  5. 질의 처리

  - 질의 전처리(언어 감지/파일 경로 힌트 추출)
  - 질의 임베딩 생성 후 OpenSearch kNN 검색
  - optional: BM25 + kNN 하이브리드
  - 상위 K개 context를 Bedrock에 전달해서 답변 생성
  - 답변에 source는 repo, file, line, commit, score 같이 항상 첨부

  ## 5) 최소 MVP 순서 (우선 구현 권장)

  2. 전체 코드 1회 인덱싱

  ## 6) 위험 요소/주의점

  - 임베딩 모델 차원 불일치(embedding mismatch) 오류
  - 너무 큰 파일 단일 청크로 인한 맥락 손실
  - 과도한 삭제/재인덱싱 비용(매일 전체 재인덱싱 금지, 증분 위주)
  - 비밀정보(키/토큰, 비공개 코드) 유출 방지: 접근 제어 + 감사 로깅 필수
  - Bedrock 호출 비용/지연: 캐시와 top-k 축소로 조절

• 이 프로젝트는 이미 얇은 RAG 골격이 있습니다. 현재 기준점은 이렇습니다.

  - app/loader.py 가 vector_store.json을 읽어 메모리에서 cosine search를 합니다.
  - app/chunker.py 는 문자 수 기준 청킹만 하고, 코드 구조나 줄 번호 정확도는 약합니다.
  - app/embedder.py 는 Bedrock 임베딩과 생성 호출을 한 클래스에 같이 넣어둔 상태입니다.
  - app/agent/service.py 는 search_repo와 retrieve_docs를 라우팅하고, app/main.py 는 /agent 하나만 노출합니다.

  즉, 방향은 새 시스템을 갈아엎는 게 아니라, 로컬 JSON 벡터 저장을 OpenSearch 인덱스로 바꾸고, 정적 코드 검색을 Git diff + Bedrock 작업까지 확장하는 쪽이 맞습니다.

  권장 설계

  1. 저장소 계층을 먼저 분리합니다. load_vector_store() 같은 파일 기반 접근을 직접 쓰지 말고, ChunkRepository 인터페이스 뒤에 JsonChunkRepository와 OpenSearchChunkRepository를 둡니다. 이렇게 해야 /agent 흐름을 거의 안 깨고 교체
     할 수 있습니다.
  2. 청크 스키마를 먼저 고정합니다. 최소 필드는 chunk_id, repo, branch, commit, file_path, language, start_line, end_line, content, content_hash, embedding 입니다. 지금처럼 start/end를 문자 오프셋으로만 두면 소스 참조 품질이 떨
     어집니다.
  3. 코드 청킹은 처음부터 AST에 집착하지 말고, 1차는 줄 번호 기반 윈도우 + 함수/클래스 헤더 휴리스틱으로 갑니다. 이 프로젝트가 여러 언어를 다룰 가능성이 있어서, 초기엔 범용 전략이 낫습니다. 현재 app/chunker.py 는 fallback 로직으
     로만 남기는 게 좋습니다.
  4. OpenSearch 인덱스는 두 개로 나눕니다. code_chunks 와 git_diff_chunks 입니다. 코드 청크와 diff hunk를 한 인덱스에 섞으면 필터링과 프롬프트 조립이 지저분해집니다.
  5. 검색 경로는 하이브리드로 갑니다. 벡터 검색만 쓰지 말고 file_path, symbol, keyword 조건을 같이 태웁니다. 현재 app/agent/service.py 의 search_repo() 문자열 매칭은 여기에 흡수되면 됩니다.
  6. Bedrock 역할을 둘로 분리합니다. 임베딩용 클라이언트와 생성용 클라이언트를 분리하고, region/model/env 도 분리합니다. 지금 app/embedder.py 는 책임이 섞여 있어서 운영 시점에 바로 불편해집니다.
  7. Git diff 인덱싱은 별도 파이프라인으로 만듭니다. git diff base..head 나 git show commit 에서 hunk 단위로 끊고, 각 hunk를 임베딩해서 git_diff_chunks 에 넣습니다. 질의가 “최근 뭐 바뀌었어”, “이 변경 리뷰해줘” 류면 diff 인덱스
     를 먼저 보고, 필요하면 관련 코드 청크를 2차로 가져옵니다.
